🎫 Gestion de Tickets

Organiser et suivre efficacement vos projets - BTS SIO SLAM

🎯 Introduction : Pourquoi gérer les tickets ?

💡 Comprendre l'importance de la gestion de tickets ▼

🎯 Objectif : Comprendre comment les tickets permettent d'organiser, suivre et prioriser le travail dans un projet de développement.

📋 Qu'est-ce qu'un ticket ?

Un ticket est une unité de travail qui décrit quelque chose à faire dans un projet :

  • Bug à corriger : "Le bouton de connexion ne fonctionne pas"
  • Fonctionnalité à développer : "Ajouter un système de commentaires"
  • Tâche à réaliser : "Rédiger la documentation API"
  • Amélioration : "Optimiser le temps de chargement de la page d'accueil"

⚡ Pourquoi utiliser des tickets ?

  • Organisation : Ne rien oublier, tout est tracé
  • Priorisation : Savoir ce qui est urgent vs important
  • Collaboration : Qui fait quoi ? Quel est l'avancement ?
  • Historique : Tracer les décisions et les changements
  • Communication : Centraliser les discussions sur un sujet précis
🎯 Analogie : Les tickets sont comme des post-it organisés - chacun représente une tâche à faire, mais avec plus d'informations (qui, quand, pourquoi, comment).

💼 Dans le monde professionnel

En entreprise, 100% des projets utilisent un système de tickets pour gérer leur travail. C'est un standard universel que vous devez maîtriser !

ℹ️
Lien avec Scrum : Vous connaissez déjà les concepts de sprint, backlog et user stories. Les tickets sont la façon concrète de représenter ces éléments dans un outil !

📝 Les différents types de tickets

💡 Comprendre les catégories de tickets ▼

🎯 Objectif : Savoir identifier et classer les différents types de travail dans un projet.

Chaque ticket appartient à une catégorie qui définit sa nature. Voici les 4 types principaux que vous utiliserez le plus souvent :

📌 Important : La catégorisation aide à prioriser et organiser le travail. Un bug critique sera traité avant une amélioration mineure.
🐛

Bug

Définition : Quelque chose ne fonctionne pas comme prévu


Exemples :

  • • Le bouton "Envoyer" ne répond pas
  • • Erreur 500 sur la page profil
  • • Les images ne s'affichent pas

Priorité : Souvent haute ou critique

✨

Feature (Fonctionnalité)

Définition : Nouvelle fonctionnalité à développer


Exemples :

  • • Ajouter un système de filtres
  • • Créer une page de statistiques
  • • Implémenter l'authentification 2FA

Priorité : Variable selon le besoin

📋

Tâche (Task)

Définition : Travail à réaliser (non technique ou technique)


Exemples :

  • • Rédiger la documentation
  • • Configurer l'environnement de test
  • • Préparer la présentation client

Priorité : Dépend du contexte

🚀

Amélioration (Enhancement)

Définition : Optimisation d'une fonctionnalité existante


Exemples :

  • • Améliorer les performances de recherche
  • • Rendre l'interface plus intuitive
  • • Ajouter des raccourcis clavier

Priorité : Souvent moyenne ou basse

📊 Autres catégories que vous rencontrerez

  • Spike / Research : Investigation technique avant de décider comment implémenter quelque chose
  • Epic / User Story : Regroupement de plusieurs tickets liés (ex: "Système de paiement" contient 10 tickets)
  • Hotfix : Correction urgente à déployer immédiatement en production
  • Technical Debt : Refactoring pour améliorer la qualité du code
⚠️
Attention : La catégorisation peut varier selon les équipes. L'important est d'être cohérent dans votre projet. Définissez vos catégories en début de projet et tenez-vous-y !

📊 Gestion de tickets dans Trello

💡 Trello : Tableaux Kanban visuels ▼

🎯 Objectif : Utiliser Trello pour organiser vos tickets de façon visuelle avec la méthode Kanban.

📋 Qu'est-ce que Trello ?

Trello est un outil de gestion de projet basé sur des tableaux, listes et cartes. Chaque carte = un ticket !

✨ Points forts de Trello

  • Simplicité : Interface intuitive, apprentissage rapide
  • Visuel : Vue Kanban claire de l'avancement
  • Flexibilité : S'adapte à tous types de projets
  • Collaboration : Partage facile avec l'équipe
  • Gratuit : Version gratuite très complète
🎯 Analogie : Trello est comme un tableau blanc avec des post-it que vous pouvez déplacer d'une colonne à l'autre pour montrer la progression.

🏗️ Structure d'un projet Trello

📝 Backlog

Toutes les idées et tâches à faire

🎯 À faire

Tickets priorisés pour le sprint

⚙️ En cours

Tickets en développement

✅ Terminé

Tickets complétés

🎫 Anatomie d'une carte Trello (ticket)

  • Titre clair : "🐛 Corriger le bug de connexion" ou "✨ Ajouter filtre par date"
  • Description détaillée : Contexte, objectif, critères d'acceptation
  • Labels/Étiquettes : Bug (rouge), Feature (bleu), Priorité haute (orange)
  • Membres assignés : Qui est responsable du ticket
  • Date d'échéance : Deadline du ticket
  • Checklist : Sous-tâches à cocher
  • Pièces jointes : Screenshots, maquettes, documents
  • Commentaires : Discussion sur le ticket

💡 Bonnes pratiques Trello

✅ À faire

  • ✓ Utiliser des emojis pour identifier rapidement les types (🐛 bug, ✨ feature)
  • ✓ Limiter le nombre de cartes "En cours" (max 2-3 par personne)
  • ✓ Ajouter une checklist dans chaque carte avec les étapes
  • ✓ Déplacer les cartes régulièrement pour refléter l'avancement
  • ✓ Archiver les cartes terminées pour garder un tableau propre
  • ✓ Utiliser des couleurs cohérentes pour les labels
💡
Astuce Pro : Créez des templates de cartes pour les types récurrents (bug report, nouvelle feature). Cela vous fera gagner du temps et assurera la cohérence !
🎓
Pour aller plus loin : Trello propose des Power-Ups (extensions) comme Calendar, Custom Fields, ou Voting pour enrichir vos tableaux.

📚 Gestion de tickets dans Notion

💡 Notion : Base de données flexible et puissante ▼

🎯 Objectif : Utiliser Notion pour créer une base de données de tickets avec plusieurs vues et propriétés personnalisées.

📚 Qu'est-ce que Notion ?

Notion est un outil tout-en-un qui combine wiki, base de données et gestion de projet. Plus puissant mais aussi plus complexe que Trello.

✨ Points forts de Notion

  • Flexibilité : Créez exactement la structure dont vous avez besoin
  • Vues multiples : Tableau, Kanban, Calendrier, Timeline
  • Propriétés riches : Select, Multi-select, Formules, Relations
  • Documentation intégrée : Pages imbriquées pour la doc projet
  • Collaboration : Commentaires, mentions, historique
🎯 Analogie : Si Trello est un tableau de post-it, Notion est un classeur Excel surpuissant avec des vues personnalisables.

🏗️ Structure d'une base de données tickets Notion

📊 Propriétés essentielles d'un ticket :

  • Titre (Text) : Nom du ticket
  • Type (Select) : Bug, Feature, Tâche, Amélioration
  • Statut (Select) : Backlog, À faire, En cours, En test, Terminé
  • Priorité (Select) : Critique, Haute, Moyenne, Basse
  • Assigné à (Person) : Qui travaille dessus
  • Sprint (Select) : Sprint 1, Sprint 2, etc.
  • Date limite (Date) : Deadline du ticket
  • Estimation (Number) : Temps estimé en heures
  • Tags (Multi-select) : Frontend, Backend, Design, etc.

👁️ Les différentes vues Notion

📊

Vue Tableau

Tableau classique avec lignes et colonnes. Idéal pour avoir une vue d'ensemble avec toutes les propriétés visibles.


Usage : Consulter rapidement tous les détails

📋

Vue Kanban

Cartes organisées en colonnes par statut. Comme Trello ! Parfait pour visualiser le flux de travail.


Usage : Gestion quotidienne du sprint

📅

Vue Calendrier

Tickets affichés sur un calendrier selon leur date limite. Super pour gérer les deadlines.


Usage : Planification temporelle

📈

Vue Timeline

Vue Gantt pour voir les dépendances et la chronologie du projet sur une ligne de temps.


Usage : Vision long terme du projet

💡 Bonnes pratiques Notion

✅ À faire

  • ✓ Créer des templates de tickets pour chaque type (Bug, Feature, etc.)
  • ✓ Utiliser les formules pour calculer automatiquement (ex: charge totale du sprint)
  • ✓ Relier les tickets à d'autres bases (ex: base "Sprints", base "Membres")
  • ✓ Ajouter des filtres et tris personnalisés dans chaque vue
  • ✓ Utiliser les sous-pages pour la documentation détaillée
  • ✓ Configurer des rappels pour les deadlines importantes

🆚 Trello vs Notion : Quand utiliser quoi ?

📊 Utilisez Trello si :

  • ✓ Besoin de simplicité et rapidité
  • ✓ Projet court ou équipe petite (2-5 personnes)
  • ✓ Workflow simple (Kanban classique)
  • ✓ Apprentissage rapide nécessaire
  • ✓ Focus sur le visuel et la clarté

📚 Utilisez Notion si :

  • ✓ Projet complexe avec beaucoup de données
  • ✓ Besoin de vues multiples
  • ✓ Documentation importante à gérer
  • ✓ Relations entre tickets nécessaires
  • ✓ Reporting et analyses avancés
💡
Conseil : Pour débuter, commencez avec Trello pour sa simplicité. Une fois à l'aise, explorez Notion pour des projets plus complexes nécessitant plus de structure et de données.

🏢 Introduction à GLPI

💡 GLPI : Outil professionnel de ticketing IT ▼

🎯 Objectif : Découvrir GLPI, un outil professionnel utilisé par les équipes IT pour la gestion des incidents et demandes.

🏢 Qu'est-ce que GLPI ?

GLPI (Gestion Libre de Parc Informatique) est un logiciel open source de gestion de services IT. Il est très utilisé dans les entreprises pour :

  • 🎫 Gérer les tickets de support (incidents, demandes)
  • 💻 Inventorier le matériel informatique
  • 📋 Suivre les contrats et licences
  • 📊 Générer des rapports et statistiques
🎯 Différence clé : Trello/Notion sont pour la gestion de projet de développement. GLPI est pour le support IT et la gestion d'incidents (helpdesk).

🌟 Pourquoi connaître GLPI ?

  • Standard professionnel : Utilisé par de nombreuses entreprises et administrations
  • Compétence demandée : Souvent recherché dans les offres d'emploi IT
  • Processus ITIL : Basé sur les meilleures pratiques de gestion des services IT
  • Gratuit et open source : Vous pouvez l'installer pour apprendre

🎫 Types de tickets dans GLPI

🚨

Incident

Définition : Interruption non planifiée ou dégradation de service


Exemple : "Mon ordinateur ne démarre plus", "Impossible d'accéder au serveur"


Objectif : Restaurer le service normal le plus vite possible

📝

Demande

Définition : Demande de service, information ou changement


Exemple : "J'ai besoin d'un nouveau logiciel", "Changer mon mot de passe"


Objectif : Répondre à un besoin utilisateur

🔧

Problème

Définition : Cause racine d'un ou plusieurs incidents


Exemple : "Le serveur redémarre sans arrêt" (cause de plusieurs incidents)


Objectif : Trouver et résoudre la cause sous-jacente

🔄

Changement

Définition : Modification planifiée de l'infrastructure


Exemple : "Migration du serveur", "Mise à jour du système"


Objectif : Gérer le changement de manière contrôlée

🔄 Cycle de vie d'un ticket GLPI

📥 Nouveau

Ticket créé

🎯 Assigné

Attribué à un technicien

⚙️ En cours

Traitement du ticket

⏸️ En attente

Attend retour utilisateur

✅ Résolu

Solution appliquée

🔒 Clos

Validé et fermé

📊 Concepts clés GLPI

  • SLA (Service Level Agreement) : Délai maximum de résolution selon la priorité
  • Catégories : Matériel, Logiciel, Réseau, Accès... pour classer les tickets
  • Priorité : Calculée automatiquement (Impact × Urgence)
  • Escalade : Transmission à un niveau supérieur si non résolu à temps
  • Base de connaissances : Solutions réutilisables pour problèmes courants
  • Suivi : Historique de toutes les actions sur le ticket
💡
Pour votre culture : GLPI est basé sur ITIL (Information Technology Infrastructure Library), un référentiel de bonnes pratiques pour la gestion des services IT utilisé mondialement.
🎓
En résumé : Vous n'utiliserez pas GLPI pour vos projets de développement étudiants, mais vous le rencontrerez probablement en stage ou en entreprise pour gérer le support utilisateur !

🔄 Intégration des tickets dans Scrum

💡 Comment les tickets s'intègrent dans un sprint Scrum ▼

🎯 Objectif : Comprendre comment organiser vos tickets dans le cadre d'une méthodologie Scrum.

🔄 Rappel : Les bases de Scrum

Vous connaissez déjà ces concepts :

  • Product Backlog : Liste priorisée de toutes les fonctionnalités à développer
  • Sprint : Itération de 1-4 semaines pour développer un incrément
  • Sprint Backlog : Tickets sélectionnés pour le sprint en cours
  • User Story : Fonctionnalité exprimée du point de vue utilisateur
🎯 Le lien : Chaque élément du backlog (user story, bug, tâche) devient un ticket dans votre outil (Trello/Notion). Les tickets sont la représentation concrète de votre backlog !

📋 De la User Story au Ticket

📝 User Story (Concept Scrum)

Format : "En tant que [utilisateur], je veux [action] afin de [bénéfice]"

Exemple : "En tant qu'utilisateur, je veux pouvoir filtrer les produits par prix afin de trouver rapidement ce qui correspond à mon budget"

🎫 Ticket (Implémentation pratique)

Titre : "✨ Ajouter filtre par prix"

Description : User story + critères d'acceptation + spécifications techniques

Checklist :

  • ☐ Design UI du filtre
  • ☐ API backend
  • ☐ Intégration frontend
  • ☐ Tests

🔄 Workflow Scrum avec tickets

1️⃣ Product Backlog

Tous les tickets du projet, priorisés par le Product Owner

2️⃣ Sprint Planning

Sélection des tickets pour le sprint (Sprint Backlog)

3️⃣ Sprint

Déplacement des tickets : À faire → En cours → Terminé

4️⃣ Sprint Review

Démonstration des tickets terminés

5️⃣ Sprint Retro

Amélioration du processus de gestion des tickets

🎯 Organisation des tickets dans un sprint

📊 Structure recommandée dans Trello/Notion :

  • Product Backlog : Colonne/Vue avec TOUS les tickets, triés par priorité
  • Sprint Backlog : Colonne/Vue avec tickets du sprint actuel uniquement
  • En cours (WIP Limit) : Max 2-3 tickets par personne en même temps
  • À vérifier / En test : Tickets terminés mais pas encore validés
  • Done (Definition of Done) : Tickets complètement terminés selon vos critères

📈 Estimer les tickets (Story Points)

1️⃣

1-2 points

Très simple

Quelques heures de travail. Ex: Changer un texte, corriger une faute de frappe

3️⃣

3-5 points

Moyen

1-2 jours de travail. Ex: Ajouter un formulaire simple, corriger un bug classique

8️⃣

8-13 points

Complexe

3-5 jours de travail. Ex: Nouvelle fonctionnalité complète avec API et UI

❓

20+ points

Epic - À découper !

Trop gros pour un sprint. Il faut le diviser en plusieurs tickets plus petits

💡 Bonnes pratiques tickets + Scrum

✅ Règles d'or

  • ✓ Un ticket = une personne : Assignez toujours un responsable unique
  • ✓ Tickets "Sprint-ready" : Description complète AVANT le sprint planning
  • ✓ Daily Scrum : Mettre à jour les tickets tous les jours
  • ✓ Pas de modification en sprint : Le Sprint Backlog est figé (sauf urgence)
  • ✓ Definition of Done claire : Quand considère-t-on un ticket vraiment terminé ?
  • ✓ Retrospective : Améliorer le système de tickets à chaque sprint
⚠️
Erreur fréquente : Commencer un sprint avec des tickets mal définis. Résultat : perte de temps en réunions pour clarifier en cours de sprint. Investissez du temps dans le "grooming" du backlog AVANT le sprint !
🎯
Métrique utile : La vélocité = nombre de story points complétés par sprint. Après 2-3 sprints, vous pouvez prédire combien de tickets vous pouvez réaliser dans les prochains sprints !

⭐ Bonnes pratiques de rédaction de tickets

💡 Comment rédiger un bon ticket ▼

🎯 Objectif : Savoir rédiger des tickets clairs, complets et actionnables.

🎯 Qu'est-ce qu'un bon ticket ?

Un bon ticket doit être SMART :

  • Spécifique : Description précise et sans ambiguïté
  • Mesurable : Critères d'acceptation clairs
  • Actionnable : On sait quoi faire concrètement
  • Réaliste : Faisable dans le temps imparti
  • Temporisé : Deadline définie
🎯 Règle d'or : Un ticket doit pouvoir être compris et réalisé par quelqu'un d'autre que vous, même 6 mois après sa création !

📝 Anatomie d'un ticket parfait

❌ Mauvais exemple

Titre : Problème connexion

Description : Ça marche pas

Assigné à : Personne

Priorité : Non définie

⚠️ Imprécis, pas d'information utile, impossible à traiter

✅ Bon exemple

Titre : 🐛 [Login] Erreur 500 à la connexion avec email contenant un +

Type : Bug | Priorité : Haute

Description :

  • Contexte : Page de login
  • Comportement : Erreur 500 pour emails avec "+"
  • Attendu : Connexion fonctionnelle
  • Impact : 5% des utilisateurs concernés

Steps to reproduce :

  1. Aller sur /login
  2. Saisir: test+alias@email.com
  3. Cliquer "Se connecter"
  4. → Erreur 500

Environnement : Chrome 120, Windows 11

Assigné à : Jean | Deadline : 15/01/2025

✓ Clair, complet, reproductible, actionnable

📋 Template pour différents types de tickets

🐛 Template Bug Report :

  • Titre : [Composant] Description courte du problème
  • Description : Quel est le problème ?
  • Steps to reproduce : 1. ... 2. ... 3. ...
  • Comportement attendu : Ce qui devrait se passer
  • Comportement actuel : Ce qui se passe réellement
  • Environnement : OS, Navigateur, Version
  • Screenshots / Vidéo : Si applicable
  • Logs d'erreur : Messages console / serveur

✨ Template Feature Request :

  • Titre : [Action] Description de la fonctionnalité
  • User Story : En tant que [qui], je veux [quoi] afin de [pourquoi]
  • Contexte : Pourquoi cette feature est importante
  • Critères d'acceptation : Liste des conditions pour considérer la feature terminée
  • Maquettes / Wireframes : Si disponibles
  • Spécifications techniques : Contraintes / dépendances
  • Estimation : Story points ou temps estimé

✅ Règles de nommage des tickets

🏷️ Convention de nommage recommandée

  • ✓ Emoji de type : 🐛 Bug | ✨ Feature | 📋 Task | 🚀 Enhancement
  • ✓ [Composant] : Module/Page concernée (ex: [Login], [API], [Dashboard])
  • ✓ Verbe d'action : Corriger, Ajouter, Implémenter, Optimiser, Refactorer
  • ✓ Description courte : Max 60 caractères pour la lisibilité

Exemples :

  • ✓ "🐛 [Login] Corriger erreur 500 avec emails contenant +"
  • ✓ "✨ [Dashboard] Ajouter graphique de statistiques mensuelles"
  • ✓ "📋 [Doc] Rédiger guide d'installation pour développeurs"
  • ✓ "🚀 [API] Optimiser temps de réponse endpoint /users"

🎯 Les 10 commandements du ticket parfait

  • 1. Un titre clair tu écriras - Quelqu'un doit comprendre le ticket en lisant juste le titre
  • 2. Le contexte tu expliqueras - Pourquoi ce ticket existe ? Quel est le problème ?
  • 3. Des étapes reproductibles tu donneras - Pour les bugs, étapes précises pour reproduire
  • 4. Des critères d'acceptation tu définiras - Comment savoir que c'est terminé ?
  • 5. Un responsable unique tu assigneras - Pas de ticket sans assignation
  • 6. Une priorité tu établiras - Critique / Haute / Moyenne / Basse
  • 7. Des screenshots tu attacheras - Une image vaut 1000 mots
  • 8. Les dépendances tu mentionneras - Liens avec d'autres tickets si nécessaire
  • 9. Régulièrement tu mettras à jour - Avancement, blocages, découvertes
  • 10. Respectueux tu resteras - Ton professionnel même pour signaler des bugs
⚠️
Erreurs courantes à éviter :
  • ❌ Titres vagues : "Problème", "Ça marche pas", "Amélioration"
  • ❌ Pas de description : Juste un titre sans contexte
  • ❌ Mélanger plusieurs sujets : 1 ticket = 1 problème/feature
  • ❌ Pas d'assignation : Tickets orphelins que personne ne prend
  • ❌ Oublier de fermer : Tickets "fantômes" en statut ouvert alors que terminés
💡
Astuce Pro : Créez des templates de tickets dans votre outil (Trello/Notion) pour chaque type. Cela assure la cohérence et fait gagner du temps !

🎓 Synthèse et Fiche Récapitulative

🎉
Félicitations ! Vous avez maintenant toutes les bases pour gérer efficacement les tickets dans vos projets de développement. Cette compétence est essentielle en entreprise et pour vos projets de BTS !

📋 Fiche Récapitulative : Checklist du gestionnaire de tickets

✅ Avant de créer un ticket

Vérifier que le ticket n'existe pas déjà - Éviter les doublons
Déterminer le type correct - Bug / Feature / Tâche / Amélioration
Évaluer la priorité - Critique / Haute / Moyenne / Basse

✅ Pendant la rédaction du ticket

Écrire un titre clair et descriptif - [Composant] Action description
Ajouter une description complète - Contexte + objectif
Définir les critères d'acceptation - Comment savoir que c'est terminé ?
Joindre des screenshots si pertinent - Une image vaut 1000 mots
Ajouter les steps to reproduce pour les bugs - Étapes précises

✅ Gestion quotidienne

Assigner un responsable unique - 1 ticket = 1 personne
Limiter le WIP (Work In Progress) - Max 2-3 tickets en cours par personne
Mettre à jour le statut régulièrement - Refléter l'avancement réel
Commenter les blocages ou découvertes - Communication transparente
Lier les tickets dépendants - Montrer les relations

✅ Dans un contexte Scrum

Groomer le backlog avant le sprint planning - Tickets prêts
Estimer les tickets en story points - Planification réaliste
Ne pas modifier le Sprint Backlog en cours de sprint - Stabilité
Démo des tickets terminés en Sprint Review - Transparence

✅ Clôture d'un ticket

Vérifier que tous les critères d'acceptation sont remplis - Definition of Done
Tests effectués et passés - Qualité assurée
Code reviewé si applicable - Validation par les pairs
Documentation mise à jour si nécessaire - Traçabilité
Déployé en production ou merged dans la branche principale - Livraison
